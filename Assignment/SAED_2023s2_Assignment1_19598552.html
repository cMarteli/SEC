4.1  Game    Logic

The  game    is   played  on  a  9×9 grid,  which    is  initially    empty   except  for  a  single square  at    the  centre,    called   the  “citadel”,  that  the player  must    defend. ̡̨̢̩̥̩̥̥

The  following    events   take  place  in  the game: ̡̨̢̩̥̩̥̥

  1. Every  1500    milliseconds,  a    new   robot  appears  randomly  in one  of    the  four    corners   of  the  grid  (unless all  four    corners  are    already   occupied  by  other  robots). ̡̨̢̩̥̩̥̥

    Each robot  has    the  following    characteristics: ̡̨̢̩̥̩̥̥

    • A   unique  ID.  The  first robot  is    number  1,    the   second  number  2,  etc. This  number    should  be    displayed   on  the  screen  underneath the  robot    icon  itself. ̡̨̢̩̥̩̥̥

      ̡̨̢̩̥̩̥̥

    • A    delay   value  d,  randomly  chosen from  between    500–2000  milliseconds.    See   below. ̡̨̢̩̥̩̥̥

      ̡̨̢̩̥̩̥̥
    ̡̨̢̩̥̩̥̥

  2. Each  robot  attempts  to move  to    a  new    grid   square  every  d  milliseconds. Robots  can    move  up,    down,   left  or  right  by one  square    at  a    time.   They  “prefer”  to  move towards  the    citadel,  either    horizontally   or  vertically,  but  otherwise move  randomly. ̡̨̢̩̥̩̥̥

    However,    a  robot    cannot   move  into  a  grid square  already    occupied  by    another   robot,  and  it  cannot move  outside    the  grid.    If   a  robot  is  completely boxed-in,  then    they  don’t    move,   and  wait  another  d milliseconds  instead. ̡̨̢̩̥̩̥̥

    Since    the  value    of   d  is  different  for each  robot,    we  expect    all   robots  to  be  moving at  different    times. ̡̨̢̩̥̩̥̥

    When  a    robot   does  move,  it  doesn’t happen  instantaneously.    Each  robot    takes   400  milliseconds  to  perform the  move,    from  start    to   end,  and  this  must be  animated    in  40    millisecond   intervals  or  smaller  (or, equivalently,  at    25  frames    per   second  or  larger).  For instance,  200    milliseconds  after    the   robot  has  started  moving, it  should    be  shown    half-way   between  one  grid  square and  another. ̡̨̢̩̥̩̥̥

    While    the  robot    is   in  the  process  of moving,  it    is  considered    to   occupy  both  its  original and  new    grid  square.    No   other  robot  can  move into  either    of  those    grid   squares. ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

  3. If  any  robot  moves into  the    citadel  square,    the   game  ends,  and  the player  receives    their  final    score. ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

  4. The   player  can  click  on an  unoccupied    square  to    build   a  “fortress  wall”.  Only one  fortress    wall  can    be   built  per  2000  milliseconds, but  a    player  can    click   on  several  squares  in quick  succession    to  queue    up   wall-building  commands.  Corresponding  walls will  then    be  built    at   a  rate  of  one per  2000    milliseconds. ̡̨̢̩̥̩̥̥

    If  a    square   is  occupied  at  the time  a    wall  is    ready   to  be  built  on it,  the    command  is    ignored,   and  there  is  no delay  until    the  next    wall   can  be  built.   ̡̨̢̩̥̩̥̥

    At most  10    fortress  walls    can   be  built  at  any one  time    (not  counting    any   that  have  been  destroyed). If  the    player  tries    to   build  more  than  10 walls  (whether    or  not    they   are  queued-up),  any  excess wall-building  commands    are  ignored. ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

  5. Walls    can   be  destroyed  by  robots. A  robot    can  run    into   a  wall,  causing  (a) the  robot    to  be    destroyed   (disappearing),  and  (b)  the wall  to    be  weakened    or   itself  destroyed.  A  wall can  survive    one  impact    from   a  robot,  and  will be  destroyed    (disappearing)  on    a   2nd  impact,  obviously  with a  different    robot. ̡̨̢̩̥̩̥̥

    The  destruction    happens   at  the  end  of a  robot’s    move.  The    robot   and  the  wall  are unchanged  until    the  moment    that   the  robot  finishes  moving into  the    grid  square    containing   the  wall. ̡̨̢̩̥̩̥̥

    Two  robots  cannot simultaneously  attack    the  same    wall,   as  they  are  required (as  above)    to  keep    out   of  each  others’  squares. ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

  6. Each time  a    robot  is    created,   the  player  creates  a wall,  or    a  wall    is   impacted,  this  should  be logged  in    an  on-screen    text   area.  (Don’t  include  other events  in    this  log.) ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥
Configurability

There    are   various  hard-coded  values  here. You  may    optionally  allow    the   player  to  specify  alternate values  before    the  game    begins,   but  the  values  shown here  should    be  the    defaults   in  any  case. ̡̨̢̩̥̩̥̥

The  grid must  be    at  least    3×3,   at  an  absolute  minimum. If  either    dimension  is    an   even  number,  then  the citadel  will    be  slightly    off-centre,   but  that’s  acceptable. ̡̨̢̩̥̩̥̥

Implementing randomness (or not)

The  word “randomly”  appears    several  times    above.   If  you  prefer,  you can  instead    implement  some    sort   of  pre-determined  pattern  or logic,  as    long  as    the   full  range  of  values is  still    in  play. ̡̨̢̩̥̩̥̥

Implementing timing

There    are   various  ways  to  implement timing  requirements.    Some  Java    API   calls  that  may  help include: ̡̨̢̩̥̩̥̥

  • System.currentTimeMillis(),  which    returns  (as    a   long)  the  number  of milliseconds  elapsed    since  1970.    (Call   this  twice,  and  take the  difference,    to  determine    how   many  milliseconds  have  elapsed between  the    two  calls.) ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

  • Thread.sleep() ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

  • Timer    and   TimerTask. ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

  • ScheduledExecutorService  and  the  factory methods  in    Executors  that    create   them. ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

4.2  Status  Information

Two  pieces of  information    (both  integers)    must   be  shown  on  the screen  at    all  times: ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

  2. The  player’s    score. ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

    ̡̨̢̩̥̩̥̥

4.3  Demo  Code

There  are pre-existing  demo    apps  written    in   Java  using  Swing  and JavaFX,  available    on  Blackboard    under   “Assignments”.  It  is  recommended that  you    choose  one    of   them  as  the  starting point  for    your  application. ̡̨̢̩̥̩̥̥

Each    demo   app  sets  up  a basic  GUI    layout,  but    also   includes  an  “…Arena”  class that  displays    a  grid    on   the  screen.  It  is not  finished.    The  intention    is   for  you  to  modify this  class    according  to    your   needs,  but  the  basic logic  regarding    where  things    are   drawn  on  the  screen is  given    to  you. ̡̨̢̩̥̩̥̥

A    selection   of  image  files  are available  in    src/main/resources  for    displaying   the  various  game  objects. You  may    substitute  other    images   if  you  prefer. ̡̨̢̩̥̩̥̥
